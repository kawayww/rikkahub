# 聊天总览摘要设计

## 背景

当前右侧“总览”页直接展示消息正文的第一行。这个方案的问题是：

1. 用户看到的是原文片段，不是对这条消息的概括。
2. 一旦消息较长，第一行不一定最有代表性。
3. 预览信息每次都是现场从原文截取，没有独立缓存，不适合后续重复浏览。

本次要做的是给聊天消息增加一个持久化的“摘要”字段，并在总览页默认使用摘要展示；摘要由一个独立的小模型后台生成，和标题总结模型、聊天建议模型、压缩模型分开。

## 目标

1. 每条已提交的 `UIMessage` 都可以持久化保存一份摘要。
2. 摘要不是临时生成，而是写回消息数据本身，后续打开总览直接读取。
3. 摘要模型独立于标题总结模型，使用单独的设置项。
4. 总览页默认使用 AI 摘要模式，用户可以切换为截断模式。
5. 总览页中 assistant 消息默认占用 70% 宽度。
6. 总览中的摘要显示要按真实布局宽度处理，而不是按字符串硬切。

## 非目标

1. 不改正常聊天页的消息渲染逻辑。
2. 不把摘要 prompt 暴露给用户编辑。
3. 不把摘要做成一个需要手动触发的功能。
4. 不修改搜索、导出、FTS 的核心行为。

## 数据模型

### 摘要字段放在 `UIMessage`

摘要应当存放在 `me.rerere.ai.ui.UIMessage` 上，而不是 `Conversation` 或 `MessageNode` 上。

原因：

1. `UIMessage` 已经是消息正文的序列化单元，最适合承载“这条消息自己的摘要”。
2. `MessageNode` 里可能存在多个消息版本，用 `UIMessage` 作为缓存粒度可以自然保留分支历史。
3. `MessageNodeEntity.messages` 本身就是 `UIMessage` 的 JSON 列表，增加可空字段不需要改 Room 表结构。

建议新增字段：

```kotlin
val summary: String? = null
```

要求：

1. 旧数据必须可以直接反序列化。
2. 新字段写回后，旧版本消息仍然保持可读。
3. 摘要缺失时，UI 允许回退到原始正文预览。

### Web / DTO 同步

`MessageDto` 也要同步增加同名字段，避免 Web 接口和本地模型脱节。

## 摘要模型

### 新增独立模型槽位

设置中新增一个专用的模型配置项，名称为“聊天对话总结模型”。

它的定位和现有的标题总结模型类似，但用途只限于消息摘要，不复用标题模型，也不和标题 prompt 共用设置。

建议命名：

1. 代码字段：`conversationSummaryModelId`
2. UI 文案：`聊天对话总结模型`

### 内置 prompt

摘要 prompt 采用隐藏的内置模板，不给用户暴露编辑入口。

要求：

1. 内容简短、稳定、可重复。
2. 输出风格固定，优先清晰概括，不输出 markdown。
3. 语言尽量跟随用户当前语言或消息主语言。
4. 允许内部根据消息角色使用不同措辞，但不做用户可配置化。
5. 对于纯附件消息，prompt 要补一段简短的附件类型说明，避免摘要输入为空。

### 模型回退

如果专用摘要模型没有配置，或找不到对应 provider / model，则回退到当前聊天模型。

如果回退后仍然没有可用模型，则直接跳过摘要生成，不阻塞消息发送，不弹出干扰性的错误。

## 生成时机

摘要生成是后台任务，不是手动按钮。

### 触发点

1. 用户消息刚被写入并保存后，触发该 `UIMessage` 的摘要生成。
2. assistant 消息生成完成并保存后，触发该 `UIMessage` 的摘要生成。
3. 用户编辑消息、重新生成消息、切换分支后，新的消息版本如果没有摘要，照样自动补生成。

### 任务原则

1. 摘要任务只针对 `summary == null` 的消息版本。
2. 摘要写回时只更新对应的 `UIMessage` 版本，不影响旧分支消息。
3. 任务失败时只记日志，不打断聊天流程。
4. 需要时可以在会话重新加载时做一次轻量回填，补上遗漏摘要。

## 总览页展示

### 显示模式

总览页新增一个显示模式配置：

1. `AI 摘要`
2. `截断`

默认值为 `AI 摘要`。

当处于 `AI 摘要` 模式时：

1. 优先展示 `UIMessage.summary`。
2. 如果摘要还没生成，临时回退到原始正文的预览片段。
3. 摘要生成后，列表自动刷新成摘要内容。

当处于 `截断` 模式时：

1. 继续展示原始正文片段。
2. 仍然按总览页的宽度约束裁切，不直接显示整段消息。

### assistant 宽度规则

总览里 assistant 消息默认限制在可用宽度的 70%。

用户消息仍然保留自然宽度，但也要遵守总览页的整体内边距和可读性约束。

### 摘要行数规则

新增一个总览摘要行数配置，默认值为 `1`。

建议实现为一个小范围整数配置，范围固定为 `1..4`，优先保持总览紧凑，避免单条摘要撑得太高。

规则如下：

1. 如果摘要在当前 bubble 宽度下的实际行数小于等于目标行数，显示实际行数。
2. 如果实际行数大于目标行数，截断到目标行数。
3. 如果摘要中的某些逻辑行在 bubble 宽度下会溢出，先按真实布局宽度重新折行，再按上面的行数规则处理。

这里的“真实布局宽度”指 Compose 实际测量后的文本宽度，不是按字符数粗切。

### 渲染实现要求

总览摘要渲染需要基于真实布局测量完成，而不是只调用 `substring` 或 `split("\n")` 之后直接显示。

推荐思路：

1. 先根据屏幕和 bubble 宽度算出实际文本可用宽度。
2. 用 Compose 的文本测量结果得到视觉行。
3. 再根据行数配置决定是否截断。

这样可以自然满足“长行要先折行，再套用行数限制”的要求。

## 设置页

### `SettingModelPage`

新增“聊天对话总结模型”卡片：

1. 允许选择 `ModelType.CHAT` 模型。
2. 逻辑与标题总结模型类似，但独立存在。
3. 不提供 prompt 编辑入口。

### `SettingDisplayPage`

新增两项总览相关设置：

1. 总览显示模式：AI 摘要 / 截断
2. 摘要显示行数：默认 1 行

这两项都应写入现有 `DisplaySetting`，保持和其他显示偏好一致。

## 兼容性与持久化

### 不需要 Room 表结构迁移

因为摘要字段放在 `UIMessage` JSON 里，而不是单独新增数据库列，所以：

1. 不需要给 `MessageNodeEntity` 或 `ConversationEntity` 做 schema migration。
2. 老数据可以直接读。
3. 新版本只是在 JSON 里多写一个可空字段。

### 兼容旧数据

1. 旧消息没有摘要时，显示层自动回退。
2. 后台摘要任务会在合适时机补写，不需要用户手工清理。
3. 分支消息各自保留自己的摘要，不因为后续分支变化而覆盖旧版本。

## 错误处理

1. 摘要模型不可用时，跳过摘要生成并回退原文预览。
2. 摘要生成异常时，不阻塞消息发送和正常聊天流程。
3. 预览渲染异常时，优先保留一条能读的回退文本，不让列表空白。

## 测试

建议至少覆盖这些点：

1. `UIMessage` 新字段的序列化和反序列化兼容性。
2. 摘要生成触发后能写回对应消息版本。
3. 摘要模型缺失时的回退路径。
4. 总览默认模式为 AI 摘要。
5. 总览摘要行数限制生效。
6. assistant 70% 宽度规则生效。
7. 长文本在总览里先按真实宽度折行，再按行数截断。

## 结果形态

完成后，用户在总览页看到的将不再是原始正文第一行，而是：

1. 默认优先显示后台生成并持久化的摘要。
2. 可以切换回截断视图。
3. assistant 消息显示更紧凑、更符合列表预览的宽度约束。
